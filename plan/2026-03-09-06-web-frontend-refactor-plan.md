# Web 前端改造计划

更新时间：2026-03-09

## 目标

1. 让 `web` 端围绕题库、试题篮、组卷、预览、导出四条主流程稳定工作。
2. 题库按“大数据量可持续”的方式演进：服务端分页，前端缓冲展示，详情按需加载。
3. Web 导出能力以 `exam-service -> export-sidecar` 为唯一正式链路。
4. 明确 `web` 与 `client` 的职责分工，避免重复建设 OCR / AI / 试卷解析。
5. 补齐桌面浏览器下的响应式和布局稳定性。

## 明确边界

### Web 负责

- 题库浏览
- 题目详情查看
- 试题篮管理
- 试卷创建、编辑、预览
- 试卷导出

### Client 负责

- OCR 识别
- AI 分析与识别工作流
- 试卷解析与确认入库

### Export-sidecar 的定位

- `export-sidecar` 是内部渲染服务，不是直接面向前端的业务服务。
- 它应继续保持：
  - 无状态
  - 注册到 Nacos
  - 只接收完整导出 payload
  - 不自己承担题目查询与业务鉴权
- 正式链路固定为：
  - `web -> exam-service -> question-core-service(batch-full) -> export-sidecar`

## 当前已完成

1. 新增 `GET /api/questions/page` 并接入 `web`。
2. 新增 `GET /api/questions/{questionUuid}` 并接入题目详情。
3. `web` 已落成“后端 100 / 前端 20”的缓冲分页模型。
4. 组卷页元信息编辑、当前大题选中、无试卷点题异常已修。
5. 导出已经保留服务端文件名，并走 `exam-service` 统一出口。
6. 登出状态清理、试题篮图片分批加载已补上。

## Phase 1：把题库查询从“本地筛选”推进到“服务端条件分页”

目标：

- 让题库在大规模数据下仍然保持语义正确和交互稳定。

任务：

1. 为 `question-core-service /api/questions/page` 增加：
   - `keyword`
   - `grade`
   - `knowledge`
   - `difficulty`
   - `source`
2. 前端 `questionStore` 改为“查询条件变化即重置分页”。
3. 题库页统计改为清晰表达：
   - 当前已加载数量
   - 当前查询命中数量
   - 服务端总量
4. 组卷页左栏题库与题库主页共用同一套分页查询模型。

验收标准：

- 搜索与筛选结果不再依赖“已经滚动到哪一页”。
- 用户不会把缓冲结果误认为全库结果。

## Phase 2：完善导出工作流

目标：

- 让 Web 导出不仅“能下文件”，还具备清晰的交互和可维护的边界。

任务：

1. 在组卷页和预览页统一导出入口体验：
   - loading
   - 错误提示
   - 文件名保留
2. 为导出增加参数面板：
   - 是否包含答案
   - 答案位置
3. 在 `docs/` 中补一张导出链路图：
   - `web`
   - `exam-service`
   - `question-core-service`
   - `export-sidecar`
4. 明确 sidecar 的运维方式：
   - 内网服务
   - Nacos 注册
   - 由 `exam-service` 做重试/超时/错误翻译

验收标准：

- 用户能明确知道当前导出是否成功、失败、失败原因是什么。
- `web` 不出现直连 sidecar 的实现或计划。

## Phase 3：补齐桌面 Web 响应式与布局稳定性

目标：

- 修掉已经可见的覆盖、挤压、窄屏下不可操作问题。

任务：

1. `AppShell`
   - 顶栏允许换行或折叠
   - 小宽度下导航不再硬挤
2. `BankView`
   - 侧栏可折叠
   - 工具栏与底部试题篮栏支持换行
3. `ExamComposeView`
   - 左右双栏在窄宽度下切上下布局
   - section 行为区允许换行
   - 按钮区避免覆盖
4. `BasketView`
   - 头部与底部操作区允许换行
   - 列表项在窄屏改为纵向信息块

验收标准：

- 1280px、1024px、768px 三档宽度下核心页面可持续操作。

## Phase 4：做轻量但完整的组卷体验收口

目标：

- 把当前“已经能用”的组卷页进一步收成一个稳定工作面。

任务：

1. 为当前选中的大题增加更明确的视觉态和快捷切换。
2. 让新增大题、删题、移动题目后的滚动和焦点更自然。
3. 评估并补充题型管理入口。
4. 让试卷预览、组卷编辑、导出参数共享同一份试卷上下文。

验收标准：

- 组卷主流程从“能工作”提升到“连续操作不别扭”。

## 推荐执行顺序

1. Phase 1
2. Phase 2
3. Phase 3
4. Phase 4

## 不再纳入 Web 改造范围

1. OCR 上传与识别工作流
2. AI 识别主流程
3. 试卷解析页面与确认入库流程

说明：

- 这些能力继续由 `client` 承接。
- `web` 如果保留相关半成品代码，应以“删除或隐藏”为优先，而不是继续补半套实现。
