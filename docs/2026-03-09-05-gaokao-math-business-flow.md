# 高考数学题库业务流程

更新时间：2026-03-09

本文只描述当前确定的两条正式业务流程，不讨论已经废弃的旧思路。

对应最终设计见：

- [2026-03-09-09-gaokao-math-final-solution.md](c:/Users/kamil.liu/source/repos/Demo0/plan/2026-03-09-09-gaokao-math-final-solution.md)

## 1. 总体原则

系统里同时存在三种状态：

1. 录入草稿态
2. 高考数学正式语料态
3. 正式组卷运行态

它们之间的边界如下：

1. 整卷 OCR 结果先进入草稿态。
2. 只有前端确认后，题目才进入高考数学正式语料态。
3. 只有当用户明确要加入正式组卷链路时，题目才被物化到现有正式题库。
4. 单题拍照查询默认只是一种临时查询，不进入任何业务主表。

## 2. 参与服务

### `gaokao-corpus-service`

负责：

- 整卷录入
- 草稿保存
- 正式发布
- 推荐查询
- 与正式题库桥接

### `ocr-service`

负责：

- OCR
- 版面识别
- 数学公式识别
- 分题辅助

### `gaokao-analysis-service`

负责：

- 文本清洗
- XML 转换
- 标签和难度分析
- 向量构建
- 推荐重排
- RAG 推荐理由

### `question-core-service`

负责：

- 承接被物化后的正式题目

### `exam-service`

负责：

- 试题篮
- 组卷
- 导出前编排

### `export-sidecar`

负责：

- 导出渲染

## 3. 流程 A：整套试卷录入

## 3.1 发起录入

1. 管理员上传一整套高考数学试卷。
2. `gaokao-corpus-service` 创建录入会话。
3. 原始 PDF / 图片保存到对象存储。
4. 会话状态进入 `UPLOADED`。

## 3.2 OCR 与分题

1. 前端触发 `OCR + 分题`。
2. `gaokao-corpus-service` 调用 `ocr-service`。
3. `ocr-service` 返回：
   - 页全文
   - 版面块
   - 数学公式
   - 分题候选区域
4. `gaokao-corpus-service` 组装出整卷草稿：
   - 草稿试卷
   - 草稿 section
   - 草稿题
   - 草稿选项
   - 草稿答案
   - 草稿资源
5. 会话状态进入 `SPLIT_READY`。

此时重要规则：

1. 允许某些题没有答案。
2. 允许题型或题号不准确。
3. 允许 XML 还未完全规范。
4. 不进入正式题库。

## 3.3 前端人工校正

1. 前端一次性渲染整卷草稿。
2. 用户修正：
   - 卷名
   - 年份
   - section
   - 题号
   - 题干
   - 选项
   - 答案
   - 图片归属
   - 子题结构
3. 前端可以多次保存。
4. 每次保存只更新草稿层，不触发正式发布。

## 3.4 AI 分析预览

1. 前端可以对单题发起分析。
2. 前端也可以对整套试卷发起批量分析。
3. `gaokao-corpus-service` 将草稿题发送给 `gaokao-analysis-service`。
4. `gaokao-analysis-service` 返回：
   - 清洗后的题干
   - XML
   - 知识点标签
   - 解法标签
   - 公式标签
   - 易错点
   - 难度值
   - 难度层级
   - 推理步数
   - 分析摘要
5. 分析结果写入草稿预览层。
6. 前端展示这些分析结果，允许用户确认或继续修改原题内容。

这里的关键不是“AI 说了算”，而是：

1. AI 负责生成候选结构化结果。
2. 前端负责最终确认。

## 3.5 正式发布

1. 用户确认整卷或单题分析结果。
2. `gaokao-corpus-service` 将已确认内容写入正式 `gk_*` 表。
3. `gaokao-analysis-service` 基于正式题构建：
   - profile
   - RAG chunk
   - question vector
   - recommend edge
4. 向量写入 Qdrant。
5. 正式题进入高考数学语料库。

发布后，该题可以被：

1. 检索
2. 推荐
3. 作为系列题候选
4. 后续物化到正式题库

## 3.6 进入正式组卷链路

1. 用户从高考数学语料库中选题。
2. `gaokao-corpus-service` 调用 `question-core-service` 创建正式题目。
3. 桥接关系被记录。
4. 之后题目进入：
   - 试题篮
   - 组卷
   - 导出
5. `exam-service` 和 `export-sidecar` 不直接读取 `gk_*` 表。

## 4. 流程 B：单题拍照检索

## 4.1 发起查询

1. 用户拍摄一道题目并上传。
2. 前端调用单题拍照查询接口。
3. 这次请求默认不创建正式业务记录。

## 4.2 OCR

1. `gaokao-corpus-service` 调用 `ocr-service`。
2. `ocr-service` 返回：
   - 题干文本
   - 如果识别到则返回答案文本
   - 题干图片或裁剪块
   - 如果识别到则返回答案图片或裁剪块

## 4.3 AI 清洗与分析

1. `gaokao-corpus-service` 将 OCR 结果发送给 `gaokao-analysis-service`。
2. `gaokao-analysis-service` 完成：
   - 题干清洗
   - XML 转换
   - 若存在答案则进行答案 XML 转换
   - 数学标签分析
   - 难度分析
   - query embedding 构建

## 4.4 相似题推荐

1. 系统用该临时题的 embedding 和结构化标签在 Qdrant 中召回候选题。
2. 结合正式题库中的 profile 做重排。
3. 生成推荐分组：
   - 同类巩固
   - 变式迁移
   - 递进提升
   - 易错辨析
4. 基于命中的 chunk 生成推荐理由。

## 4.5 返回前端

前端一次性获得：

1. OCR 结果
2. 清洗后的题干
3. 可渲染的 XML
4. 若存在则返回答案
5. 标签和难度
6. 相似题推荐结果
7. 推荐理由

这里的关键规则是：

1. 该拍照题只是本次查询的输入对象。
2. 它不进入 `gk_question`。
3. 它不进入 `q_question`。
4. 渲染格式尽量复用现有题目展示结构。

## 5. 两条流程的落库时机

## 5.1 整套试卷录入流

落库规则：

1. OCR 后先落草稿层。
2. 人工修正仍然只改草稿层。
3. AI 分析结果先写分析预览层。
4. 只有用户确认后，才进入正式语料层。
5. 只有用户明确要组卷时，才进入正式题库层。

## 5.2 单题拍照检索流

落库规则：

1. 不落高考数学正式语料层。
2. 不落现有正式题库层。
3. 仅保留必要的技术任务状态和日志。

## 6. 为什么要这样分

因为这正好对应两种完全不同的业务目标：

1. 整卷录入是在生产正式语料。
2. 单题拍照是在消费正式语料。

如果把这两条链混在一起，会出现三个问题：

1. 噪声 OCR 结果污染正式题库。
2. 临时查询题误入推荐语料库。
3. 组卷运行链路被教研语料链路拖脏。

## 7. Phase 1 必须打通的完整链路

第一阶段的最小完整能力应是：

1. 上传整套试卷
2. OCR + 分题
3. 前端整卷修正
4. 单题 / 整卷分析
5. 分析预览确认
6. 发布到正式高考数学语料库
7. 建立向量索引
8. 单题拍照查询
9. 相似题推荐
10. 物化到正式组卷链路

这就是当前项目真正应该建设的业务流程。
